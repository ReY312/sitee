# Отчет по доработке

## Что исправлено по запросу

1. Дизайн переделан под предоставленные макеты:
   - двухколоночный блок (календарь + правая карточка);
   - поля ФИО/СНИЛС появляются только после выбора дня;
   - добавлена кнопка «Записаться»;
   - добавлено модальное окно подтверждения записи.
2. Изменена логика API:
   - клиент отправляет только выбранную дату (`selectedDate`), а не конкретное время.
3. Время приема теперь определяется автоматически на сервере/в БД.

## Алгоритм ограничения заявок на одно время

На сервере добавлены переменные:
- `SLOT_CAPACITY`
- `SLOT_START_HOUR`
- `SLOT_END_HOUR`
- `SLOT_DURATION_MINUTES`

SQL-функция перебирает слоты по выбранной дате и выбирает первый, где число активных записей меньше `SLOT_CAPACITY`.
Если свободных слотов нет — возвращается ошибка.

## Безопасность

Сохранены и используются:
- CSRF token;
- проверка Origin;
- ограничение размера body;
- rate limit;
- защита от повторной активной записи по СНИЛС.

## Тестирование

- Unit-тесты валидации (включая новую валидацию даты).
- API-тесты для:
  - успешной записи,
  - ситуации «нет свободных слотов»,
  - блокировки без CSRF.


## Повторное применение PR после конфликтов

Изменения по календарю (июнь/июль 2026), fallback RPC в backend, улучшенная диагностика 500 и синхронизация `schema.sql` с фактическими параметрами записи повторно применены в ветке.


## Мобильная адаптация

- Добавлены адаптивные брейкпоинты (`960px`, `768px`, `480px`) для календаря, формы, футера и модального окна.
- Увеличены тап-зоны элементов управления (минимум 44-48px) для удобства на телефонах.
- На узких экранах блоки перестраиваются в одну колонку, а модальное окно становится mobile-friendly (нижний лист).


## Сообщение при повторной записи по СНИЛС

Если по СНИЛС уже есть активная запись, backend дополнительно запрашивает из БД `appointment_at` и возвращает понятное сообщение:
`Для данного СНИЛС уже существует запись на <дата и время>`.
